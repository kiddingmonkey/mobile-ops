# Mobile-Ops 移动运维小程序 - 需求汇总

> 版本：v1.0（MVP）
> 日期：2026-07-15
> 目标：手机上完成 K8s 扩缩容 + 监控查看的一体化运维工具

---

## 一、项目定位

**一句话描述**：一个飞书小程序，让运维在走路/地铁上单手完成 K8s 集群扩缩容决策与执行。

**核心用户**：xiaojiahuo（初期单用户，后期扩团队）

**核心痛点**：
- 外出时收到告警，看监控要电脑，操作要电脑，手机上要装 3-5 个 App 才能完成
- 期望：**告警 → 打开小程序 → AI 建议 → 一键扩容**，5-6 步搞定

---

## 二、使用场景（用户使用习惯）

### 2.1 触发场景分布

| 场景 | 占比 | 特点 |
|---|---|---|
| **收到飞书告警后被动打开** | 70% | 紧急、需要 10 分钟内响应、单手操作为主 |
| **每日主动巡检**（早上/下班前） | 30% | 常态化、可以稍微仔细看 |
| **特殊高峰期重保**（开学季等） | 高峰月份 | 紧急扩容频繁、需要快速批量操作 |

### 2.2 使用环境

| 环境 | 占比 | 交互要求 |
|---|---|---|
| **走路/地铁上（单手）** | 80% | 大按钮、下半屏操作、手势支持 |
| **坐下来仔细看** | 20% | 支持多维度对比、详细图表 |

### 2.3 业务节奏

- **暑期**：日常调整多（缩容、优化）
- **开学季**：紧急扩容多（业务突增）
- **每日业务高峰期**：需要 10 分钟内响应

### 2.4 权限模式

- **MVP**：单用户直接执行，**不做审批流**
- **V3+**：团队接入后再加审批流

## 三、数据准确性（重点）

### 3.1 实时性要求（硬性）

| 数据类型 | 延迟要求 |
|---|---|
| 扩容决策数据 | **实时**（<30s） |
| Grafana 数据 | **不接受 2 分钟前的数据** |
| 近期趋势（5-10min） | ❌ 不可接受用于决策 |

**结论**：所有决策数据必须打时间戳，超过 30s 自动重新拉取。

### 3.2 数据源策略

| 数据类型 | 主源 | 校验源 | 展示策略 |
|---|---|---|---|
| 监控指标（CPU/内存/网络） | Grafana API | Prometheus API | 优先 Grafana |
| K8s 节点资源 | Prometheus `node_cpu_usage` | `kubectl top nodes` | **交叉验证** |
| Pod 状态 | K8s API | - | 直接读 API |

**不一致时**：**都显示并标注数据源**，让用户自己判断（不做自动决策）。

例如：
```
Node-01 CPU:
  Grafana: 73% (10s ago)
  kubectl top: 68% (25s ago)
```

### 3.3 扩容前预检查（5 项）

MVP 必须实现的预检查：

- [x] **云厂商配额**是否足够
- [x] **节点组状态**（是否有 NotReady 节点）
- [x] **历史扩容记录**（5 分钟内不重复扩容）
- [x] **成本预估**（扩容后每月增加金额）
- [x] **必须通过节点池扩容**（不允许直接建单机）

### 3.4 操作安全性

- **网络断开**：操作带唯一 ID + 幂等，重复提交不重复执行
- **成功确认**：操作日志 + 状态轮询（新节点必须变 Ready 才算成功）
- **失败回滚**：扩容失败自动记录原因，缩容失败停在中间状态并告警

## 四、页面美观与实用性

### 4.1 设计风格

**飞书简洁商务风 + Grafana 深色数据风**
- 主背景：深色（`#1F2329` / `#2D3139`）
- 强调色：飞书蓝（`#3370FF`）+ 状态色（正常绿、警告黄、严重红）
- 字体：飞书默认字体 / SF Pro / 苹方
- 数据展示：借鉴 Grafana 的图表和数字高亮

### 4.2 交互流程（核心路径）

```
飞书告警卡片（带链接）
    ↓
打开小程序首页（看集群总览）
    ↓
点"智能诊断"（AI 分析告警根因）
    ↓
看 AI 建议 + 监控图（"建议扩 2 台 8C16G"）
    ↓
确认扩容内容（节点组/规格/数量/成本）
    ↓
点"一键扩容"
    ↓
确认弹窗（二次确认）
    ↓
看到扩容明细步骤（实时进度）
    ↓
完成（节点 Ready）
```

**共 7 步，但每步都必须信息足够全。**

### 4.3 单手操作设计

- 所有主操作按钮：**屏幕下半部分**
- 支持手势：下拉刷新、左滑返回、上滑加载
- 大按钮（≥ 44px）避免误触
- 危险操作（缩容）：需要滑动确认

### 4.4 快捷指令

用户可预配置常用操作：
- "prod-web 扩容 2 台"
- "prod-api 缩容 1 台"
- "一键巡检所有集群"

点击后跳过前置步骤，直接到确认页。

### 4.5 离线降级

- **默认**：每次打开实时拉取
- **网络差时**（超时 3s）：降级读缓存（最近 1 小时的监控快照 + 集群状态）
- **明确标注**："缓存数据，非实时"

### 4.6 首页信息密度

首页显示：
1. **集群健康状态**（正常/警告/严重，带图标）
2. **核心指标卡片**（CPU/内存/Pod 数 + 趋势箭头 ↑↓）
3. **最近 3 条告警**（可点击查看详情）
4. **快速操作入口**（扩容/缩容/巡检）
5. **数据时间戳**（"数据更新于 15s 前"）

## 五、工程化落地

### 5.1 技术选型

| 层级 | 技术 |
|---|---|
| **前端** | 飞书小程序（TMA 模式，Tea + Modern 前端框架） |
| **后端** | Go（Gin/Echo） |
| **数据库** | PostgreSQL（日志、配置、快照缓存） |
| **部署** | 单机（10.210.20.130，路径 `/data2/haowu33/mobile`） |

### 5.2 前端开发方式

**由 Claude 全部开发**，使用飞书小程序 TMA（Tea/Modern App）模式。

### 5.3 后端架构

```
飞书小程序
    ↓ HTTPS
Go API (Gin)
    ├── /api/auth        飞书 OAuth
    ├── /api/cluster     集群状态
    ├── /api/metrics     监控数据（聚合 Grafana + Prom + kubectl）
    ├── /api/scale       扩缩容操作
    ├── /api/precheck    预检查
    ├── /api/operation   操作日志与状态轮询
    └── /api/alert       告警接入
    ↓
├── Grafana API
├── Prometheus API
├── K8s API (kubectl / client-go)
├── 云厂商 SDK（华为/阿里/腾讯，V2 启用）
└── PostgreSQL
```

### 5.4 安全策略

| 项 | 方案 |
|---|---|
| **用户鉴权** | 飞书 OAuth 2.0 |
| **kubeconfig 存储** | 加密存储 PostgreSQL（AES-256，密钥独立管理） |
| **云厂商 AK/SK** | 加密存储 PostgreSQL |
| **操作审计** | 全量记录，保留 180 天 |
| **API 鉴权** | JWT Token（有效期 8 小时） |
| **敏感操作** | 二次确认 + 短信验证码（V2 加） |

### 5.5 部署

- **服务器**：10.210.20.130
- **路径**：`/data2/haowu33/mobile`
- **模式**：单机部署（后期需要 HA 再扩）
- **反代**：Nginx（HTTPS 必需，飞书小程序要求）
- **域名**：需要一个 HTTPS 域名（待确认是否可用 Alert Analyzer 的域名）

### 5.6 MVP 范围确认

**MVP（第一版，2 周）：**
- ✅ 飞书 OAuth 登录
- ✅ 集群总览 & 健康状态
- ✅ 监控查看（嵌入 Grafana 关键面板 + 关键指标卡片）
- ✅ K8s 节点扩缩容（基于节点池，带 5 项预检查）
- ✅ 操作日志（谁、何时、做了什么）
- ✅ 状态轮询（扩容后确认节点 Ready）
- ✅ 快捷指令
- ❌ 云厂商购买（V2）
- ❌ AI 智能诊断建议（V2）
- ❌ 审批流（V3）
- ❌ 成本分析（V3）

**排期：**
- Week 1-2：MVP 交付
- Week 3-4：V2（云厂商购买 + AI 建议）
- Week 5-6：V3（审批流 + 成本分析）

## 七、待用户补充的信息（准备开发前必需）

在开始编码前，需要用户提供：

1. **Grafana 访问信息**
   - Grafana URL
   - API Token / 用户名密码
   - 关键面板 ID（CPU/内存/网络/Pod）

2. **Prometheus 访问信息**
   - Prometheus URL
   - 认证方式（如有）

3. **K8s 集群信息**
   - 有几个集群需要管理
   - kubeconfig 文件位置（记忆里提到 Lens 目录）
   - 是否有节点池（cluster-autoscaler 或云厂商 nodepool）

4. **飞书应用**
   - 是否已有企业飞书自建应用？
   - 若无，需要创建流程指引

5. **域名与 HTTPS**
   - 10.210.20.130 是否有公网/公司内网 HTTPS 域名？
   - 是否可复用 Alert Analyzer 的域名？

6. **告警对接**
   - 现有飞书告警卡片是从哪个系统发出的？
   - 是否可以在告警卡片里加"打开小程序"按钮？

7. **云厂商信息**（V2 用，MVP 可先不给）
   - 华为云 / 阿里云 / 腾讯云？
   - 主要用哪家？
